# Loan API on Yii2

Тестовый проект API для подачи и обработки заявок на займ.

## Что реализовано

- `POST /requests` — создание новой заявки на займ.
- `GET /processor?delay=5` — обработка всех ожидающих заявок.
- Валидация входных данных.
- Запрет на подачу заявки пользователем, у которого уже есть одобренная заявка.
- Рандомное одобрение с вероятностью `10%`.
- Защита от гонок при параллельной обработке:
  - одна заявка не может быть взята в обработку дважды;
  - у пользователя не может появиться больше одной одобренной заявки.
- Автоматический запуск миграций при старте контейнеров.

## Стек

- PHP 8.2
- Yii2
- Nginx
- PostgreSQL 16
- Docker Compose

## Запуск

> В проекте используется `network_mode: host`, чтобы приложение было доступно на `localhost:80`, а PostgreSQL — на `localhost:5432` с требуемыми параметрами подключения.
> Это подходит для Linux-окружения и типичного CI/автопроверяющего стенда.

### 1. Поднять проект

```bash
docker compose up --build
```

После старта будут доступны:

- API: `http://localhost`
- PostgreSQL:
  - host: `localhost`
  - port: `5432`
  - db: `loans`
  - user: `user`
  - password: `password`

## Примеры запросов

### Создать заявку

```bash
curl -i -X POST http://localhost/requests \
  -H 'Content-Type: application/json' \
  -d '{
    "user_id": 1,
    "amount": 3000,
    "term": 30
  }'
```

Успешный ответ:

```json
{
  "result": true,
  "id": 42
}
```

Ошибка валидации:

```json
{
  "result": false
}
```

### Запустить обработчик

```bash
curl -i 'http://localhost/processor?delay=5'
```

Ответ:

```json
{
  "result": true
}
```

## Особенности реализации

### Подача заявок

При создании заявки выполняются базовые проверки:

- `user_id`, `amount`, `term` обязательны;
- все поля должны быть целыми числами больше нуля;
- у пользователя не должно быть уже существующей заявки в статусе `approved`.

### Обработка заявок

Обработчик работает в цикле, пока в базе есть заявки в статусе `pending`.

Для корректной параллельной обработки используются:

- `FOR UPDATE SKIP LOCKED` — чтобы один и тот же pending-запрос не был взят двумя процессами одновременно;
- `pg_advisory_xact_lock(user_id)` — чтобы финальное решение по заявкам одного пользователя не нарушало ограничение на единственный `approved`;
- partial unique index в PostgreSQL на `user_id` для записей со статусом `approved` как дополнительная защита на уровне БД.
